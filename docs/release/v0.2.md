# Forge v0.2 — Skills, Tool Pool, Triggers

**Released:** 2026-04-14

Forge v0.2 extends the stack with eval/skills in the engine and harness, a tool pool and context model, and trigger-based runs with a daemon, queue, and webhook API.

---

## Sub-plan A: Skills and EvalNode

- **`NodeTypeEval`** added to the engine type system
- **`EvalNode`** struct with execution logic
- YAML **`type: eval`** parsing
- **Skill types and frontmatter parser** (TypeScript)
- **Skill registry** with filesystem scan
- **Skill resolver** with keyword matching
- **Skill lifecycle:** evaluate, promote, compare
- **Skills integrated into AgentService** (optional resolution)
- **Built-in skills:** `skills/coding/implement-feature`, `skills/quality/code-review`

---

## Sub-plan B: Tool Pool and Context

- **Tool types system** (TypeScript)
- **Tool pool assembly** (pure function)
- **Deferred tool loading** with context budget
- **Tool lifecycle hooks** (pre/post)
- **Subagent context isolation**
- YAML **`depends_on`** vocabulary for node dependencies

---

## Sub-plan C: Triggers and Parallel Runs

- **RunRegistry** — concurrent-safe in-memory run state tracking
- **RunQueue** — bounded-concurrency worker pool with `PipelineExecutor` interface
- **TaskAssigner** — rule-based adapter selection (defaults to `claude`)
- **Webhook HTTP handler** — `POST`/`GET` `/api/v1/runs` with Enqueuer and StatusGetter interfaces
- **`forged` daemon** — HTTP server with signal handling and graceful shutdown
- **Integration smoke test** — webhook → queue → registry end-to-end
- **32 new tests**, all passing with `-race`

---

## Known limitations (tracked for v0.3)

- Daemon shutdown does not drain buffered queue items (a RunQueue drain/wait API is needed).
- `repo_url` is accepted in the webhook API but is not mapped to a local path.
- TaskAssigner is not yet wired into the pipeline (scaffolding for multi-adapter runs).
